# RR38P Retro Replay Wedge ‚Äî Analysis & Working Integration

Status as of this writing: **round 10 (2026-07-07, not yet hardware-tested) is a stability/root-cause round driven by hardware crash reports against the round-9 build ‚Äî four real defects found and fixed, two of them proven mechanisms for the reported "crash into the RR freeze menu" hangs.** See the round-10 section immediately below for the full analysis; rounds 8/9 remain in the build unchanged (they were investigated as suspects and cleared ‚Äî the READY-path rewrite is semantically faithful to stock `$e38b`, verified against the KERNAL documentation and by py65 simulation of the actual built ROM).

**Round 10 (2026-07-07, tenth pass) ‚Äî crash/hang root causes: cart-owned stack-page RAM, wrong-bank ILOAD/ISAVE, destructive REU probe, unbounded UCI waits. NOT YET HARDWARE-TESTED.**

Reported symptoms (2026-07-06 hardware, round-9 build): (a) `#h` ‚Üí `pwd` ‚Üí F1 hangs, while `#h` ‚Üí F1 plays the game fine; (b) after a clean reboot, `#h` ‚Üí `cd sd` (or `cd /`) dies with a "font switch" (letters become horizontal lines) and a jump into the RR freeze menu; same for `i:hi` and for `TASM` as first command; (c) C=+CTRL+3 ‚Üí C=+CTRL+2 console hop mostly hangs, with a stray `!` on screen; (d) TODO notes: `m:<query>` worked, then "started hanging with no crt or server change"; `memcpy` hangs; C=+RESTORE Ultimate menu sometimes stops working. Note the shared signature: a jump into the freeze menu is the cart's **CBINV/BRK hook** (`$0316` ‚Üí `$decd`) firing ‚Äî i.e. the CPU executed a `$00` byte somewhere; the "horizontal lines" font garbage is collateral of executing garbage before the BRK lands.

**Defect 1 ‚Äî `wc_buf` at `$0110-$014f` destroyed the cart's resident stack-page machinery (proven; fixed by moving `wc_buf` to `$0200`).** The stock firmware keeps *resident code and variables in the low stack page*: a 23-byte helper routine at **`$0120-$0136`** (installed from bank01 ROM `$8df2` by the directory-print path at `$8db7`) plus variables at `$0134/$0135`, `$0158/$0159`, **`$015e` (a saved stack pointer that gets `txs`'d!)** and `$0168`. The **fastloader calls `$0120` on every fast load** ‚Äî `$8926`'s loader body does `jsr $de9d`, and that trampoline's code is `jsr $deba / jsr $0120 / jsr $de97 / jmp $8de1` (bank01 `$9e9d`, confirmed by raw byte scan: `jsr $0120` callers exist in bank01's loader at `$892b/$8933/$893c` and in bank03's monitor). Round 6 placed `wc_buf` (the `ll`/`dir` entry buffer, also `cp`'s filename stash) at `$0110-$014f` ‚Äî right on top of it. **Every `ll`, `dir`, or `cp` overwrote the `$0120` helper; the next fastload (F1's `%0:*`, `/`, a chain load) executed the leftover directory-listing text as code ‚Üí BRK ‚Üí freeze menu.** This is the strongest candidate for symptom (a) whenever a listing had run earlier in the session, and it made F3/`$` directory printing fragile the same way (that machinery also lives there, along with the saved-SP `txs`). Fix: `wc_buf` (and the new `mcio_ram`, below) moved to the **BASIC input buffer `$0200-$0258`**, which is provably free during command dispatch (the typed line is already shadow-copied to `cf_shadow`, `wi_sniff` read `$0200` before dispatch, and BASIC only refills it on the next typed line). Hard rule recorded in both `.asm` const blocks: **`$0110-$01c0` is cart-owned territory** ($0120-$0136 code, $0134-$0168 variables, $0170-$01c0 the TASS/TMP launcher's relocation target) ‚Äî never put wedge state there.

**Defect 2 ‚Äî `memcpy`'s KERNAL LOAD/SAVE executed a wrong bank's trampoline bytes (proven; fixed with a RAM bank-bracket stub).** KERNAL LOAD/SAVE dispatch through ILOAD/ISAVE (`$0330/$0332`), which this cart keeps pointed at `$de6d/$de6f` ‚Äî addresses in the I/O1 mirror, whose content is `$9e6d/$9e6f` **of whichever bank is currently paged in**. `cmd_memcpy` runs with **bank05** paged, and bank05's `$9e6d/$9e6f` are `$00` (byte-verified: bank01 has the fastload trampoline `38 24 18 85 93 20 97 de 4c 26 89` there, bank05 has zeros) ‚Üí `jsr LOAD` from the dispatcher hit BRK ‚Üí freeze menu. That is symptom (d)'s "memcpy zatuhava" exactly. Fix: `mcio_stub` ‚Äî a 20-byte relocatable stub copied to `mcio_ram` (`$0245`, inside the input buffer, `.errorif`-guarded) whose `jsr` operand is patched to `$ffd5`/`$ffd8` by `mcio_prep_load`/`mcio_prep_save`; it pages **bank01** in around the KERNAL call (running from RAM so it survives the repage) and restores bank05 after, preserving A/X/Y/flags both ways. Verified by py65 simulation of the built ROM: both memcpy forms now reach `$de6d`/`$de6f` with bank 1 paged (previously bank 5). General lesson recorded: **any KERNAL entry that dispatches through a cart-hooked vector (`$0330/$0332`, and the conditional `$0320/$0326` CHROUT hooks the cart's own commands install) must only be called with bank01 paged in.** The dispatcher's ordinary printing is safe ‚Äî the per-line path (`bank01_sub_8385`) restores `$0326/$0320` to stock before any typed line runs ‚Äî but this constraint is now documented.

**Defect 3 ‚Äî the boot-time REU size probe wrote into the REU without restoring it (transparency violation; prime suspect for the `TASM` crash; fixed with a non-destructive power-of-two probe).** Round 7's `reu_detect` wrote a byte to **offset 0 of every 64KB REU bank** (all 255 of them on a 16MB REU) on **every cold boot**. This cart build ("tmp12reu") keeps Turbo Macro Pro's launch image *in the REU*: the `\` handler (`$9d1d`, jump table `$9d1c`) blindly DMAs REU `$0800-$ffff` into C64 RAM `$0800+` and jumps to `$8000`, and the TASS install path (`$80e7` ‚Üí cross-bank bank7 `$800f`) manages that image ‚Äî plus the user's own REU-resident data (the 3D-renderer textures, TMP work state) lives there across sessions. Poking the REU at boot both violates the "as if the shell weren't there" requirement outright and plausibly corrupts whatever the TMP/TASS machinery keys off. Fix: the probe now (1) climbs a **power-of-two rung ladder** (banks 1,2,4,‚Ä¶,128 ‚Äî real REU sizes are powers of two, so the result is still exact, and at most 9 bytes are ever touched instead of 256), and (2) **reads out and writes back every byte it touches**, including the aliased case where a rung write physically lands on bank 0. Verified by py65 simulation against REU models of every size 0/1/2/‚Ä¶/256 banks: size detected correctly in all cases and **REU content byte-identical after the probe in all cases**. (`reu_save` = `$0201-$0209`, boot-time only; `reu_probe_res` = zp `$fd`.)

**Defect 4 ‚Äî every UCI wait loop was unbounded (fixed: all bounded now).** `net_wait_idle`, `net_wait_not_busy`, `net_wait_not_busy_spin` and `net_accept`'s handshake wait were pure `lda $df1c`-poll loops with no exit. If the command interface was ever left busy or mid-handshake ‚Äî a firmware hiccup, a command aborted halfway (freeze button / Ultimate menu / C=+RESTORE landing mid-transaction, cf. the TODO note that the Ultimate menu itself sometimes stops responding), or a TCP connect to an unreachable server stalling firmware-side ‚Äî the *next* wedge command spun forever with interrupts off: dead cursor, dead keyboard, only the freeze button left. That is the exact shape of symptom (d)'s "`m:` worked, then started hanging with **no crt or server change**" (state-dependent, not a code regression), and the most plausible mechanism for (b)'s `cd sd`/`i:hi` post-reset freezes and (c)'s console-hop hangs ‚Äî with the freeze-menu jump being the user's own freeze-button escape (or a subsequent BRK) rather than the command itself crashing. Now: idle wait ~3s, not-busy wait ~6s (DOS mounts are legitimately slow), the spinner wait ~35s with carry-set-on-timeout (a `SOCKET_READ` stays busy firmware-side until the LLM reply arrives ‚Äî this is where that latency shows up; `net_read_response` treats the timeout as a failed read), accept handshake ~1s. On timeout they simply return ‚Äî every wait is followed by push/status/accept checks that already surface errors (`?ERR`, "NOT REACHABLE", silent chat fail), so the worst case is now a few seconds of stall and a clean READY instead of a dead machine. New 2-byte countdown `w_to` at `$03fc/$03fd` (past the datassette buffer, plain unused RAM; a console-switch IRQ nesting into a foreground wait can stretch a timeout, never hang it). Also fixed while in there: `wd_chat`/`send_cli` left the TCP socket open when the write or read failed ‚Äî they now close it (`wdc_fail_close`), so a failed chat doesn't leak sockets into the firmware.

**Cleared as suspects (investigated, no change needed):** round 8's IERROR-stub bit7 rewrite and `print_ready` (byte-verified both template copies identical in the built image; simulated end-to-end with py65 ‚Äî every dispatch path including the boot `X=$80` pass re-enters IMAIN cleanly; stock `$e38b` semantics confirmed from the KERNAL doc: bit7 prints READY and continues through `($0302)`, which is exactly what the wedge does); round 9's `bank03_api_11` neuter (the cross-bank RTI-trick return path works identically for a plain `rts`); the ie-stub's hardcoded `lda #$08` bank restore (correct **by design**, not an oversight: the stub's tail `jmp ($0302)` lands in `$de8f`, whose trampoline code exists only in bank01's mirror ‚Äî byte-scan confirmed `$9e59-$9eff` is *not* shared across banks, bank05's `$9e8f` is zeros ‚Äî so bank01 must be forced there; errors/warm-starts can only legitimately fire from bank01 context since MON/TASS/RUN/SYS lines disarm the hooks first); `boot_call`'s placement at bank01 `$9e3a-$9e58` (stock bank01 had zeros there; only bank0 has code at those offsets, and it executes at `$fe3a+` in freeze mode, not via the `$de3a+` mirror).

**Round 9 (2026-07-06, ninth pass) ‚Äî undo round 8's forced light blue: the shell must never touch text color. NOT YET HARDWARE-TESTED.**

- **User feedback, verbatim** (a TODO.md note the user pointed at, not a chat message): *"Ragedring the READY. color: do not change the color of the READY. prompt to light blue and do not change it back to white after. Just do not touch the colors at all. For example if the user decides to change cursor color to red then anything that will be printed, include the READY. prompt is expected to be red."*
- **Diagnosis:** round 8 fixed the *symptom* (forced white) by forcing a different color (light blue) in `print_ready`, rather than removing the thing that forces white in the first place. That thing is **bank03 `bank03_api_11`** (`$9f8e-$9f9d`): `lda $d021 / and #$0f / ldx #$01 / cmp #$01 / bne .. / dex / stx $0286 / rts` ‚Äî reads the border color and stores either 0 (black) or 1 (white) into `$0286` (current text color), i.e. it always forces black-or-white, never anything the user picked. Confirmed via cross-reference: the *only* caller is **bank01 `bank01_sub_83ca`** (a `bank01_sub_9f51` cross-bank call encoding target `$802a` = this routine's jump-table slot, `bank03_api_11`), which itself is called from exactly two places in bank01 ‚Äî `bank01_api_00` (cold start, right after the boot banner) and `bank01_sub_83d8` (the STOP+RESTORE/warm-start re-init path). Both cold boot and RESTORE were therefore forcing this color, independent of anything the wedge did.
- **The fix:**
  - **bank03:** `bank03_api_11` neutered to a plain `rts` followed by 15 NOPs (same 16-byte footprint, nothing after it shifts). It is never called anywhere else, so this is safe with no other side effects ‚Äî text color is now simply never touched by this routine, on cold boot or RESTORE.
  - **bank05:** `print_ready` no longer does `lda #$0e / sta $0286` before printing. It still prints capitals `READY.` (unchanged from round 8) and still replicates stock `$a474`'s SETMSG side effect (`lda #$80 / jsr $ff90`), but color is left exactly as it was ‚Äî whatever the user (or nothing) last set it to.
  - Net effect: color is now **never** written by any of this project's code, anywhere. The C64's own default (light blue, from KERNAL's `$ff5b`/CINT at boot) is what shows unless the user changes it themselves (e.g. `POKE 646,2` for red), and that choice now persists through READY, the cursor, and all wedge/chat output, exactly as stock BASIC behaves.
- **Build verified:** rebuilt clean; diff-localized to bank03 `$9f8e-$9f9d` (16 bytes, the neutered routine ‚Äî confirmed byte-for-byte `60 ea ea ea ea ea ea ea ea ea ea ea ea ea ea ea` in the rebuilt image) plus the unrelated, unchanged `$9fc8-$9fca` (round-7's version-line suppression); bank05's `print_ready` shrank by 2 bytes internally (no `.errorif`-guarded structures affected). Banks 00/01 untouched this round.
- **Hardware retest checklist:** (1) boot: READY still capitals, but now in the *user's actual current color* ‚Äî on an unmodified setup that's the C64 default light blue, same visual result as round 8 claimed, but for the right reason; (2) `POKE 646,<n>` (or equivalent cursor-color change) before/after a command, confirm READY/cursor/output all pick up `<n>`, not forced back to light blue or white; (3) STOP+RESTORE: confirm color still isn't forced back to white afterward; (4) full round-8 regression (banner, wedge commands, chat-forward, freeze menu, console switching, game-transparency) ‚Äî none of that logic changed this round, only the color write was removed.

Fixing that would need BASIC-in-RAM or a CHROUT hook; deliberately out of scope this round.
- **Cross-bank call mechanism, now fully decoded (was partially understood in round 7):** `bank01_sub_9f51` + a 2-byte inline word = "call absolute address <word> in bank03". It pulls its own return address, re-pushes it +2 (skipping the inline word), pushes `$08/$de/$e2` (restore-bank01-and-return path via the `$dee3` mirror), pushes the inline target word, saves A/flags, loads the bank value from the `$deee,x` table (x=3 from the `jsr` operand low byte `$51 - $4e` ‚Äî the 5 NOPs at `$9f51` make `$9f51..$9f55` valid entry points selecting banks 3..7), and ends `jmp $dede`: the mirror-page trampoline (`sta $de00` / `pla` / **`rti`**) ‚Äî RTI, not RTS, so the pushed target is used *exactly*, no +1. This is how the cold-start tail calls bank03 `$8048` (‚Üí the version-line print, round-7-patched to a no-op) and how `bank01_sub_83ca` calls bank03's white-setter. The tail's final hand-off (`lda #$e3 / pha / lda #$85 / sta $0313 / pha` ‚Üí an eventual RTS lands at `$e386`) is left untouched ‚Äî note `$0313` is *read back* by the warm-start path (`$83e1: lda $0313 / bne`), one more reason not to repurpose those bytes.

**Round 5 (2026-07-04, fifth pass) ‚Äî new `memcpy` command, plus a `cp`/CSDB bugfix found while documenting:**
- **`memcpy $start-$end filename`** saves the inclusive memory range `$start`-`$end` to `filename`; **`memcpy filename $start`** loads `filename` into memory starting at the explicit address `$start`, **relocatably** ‚Äî any load address embedded in the file itself is ignored. Implemented in `rr38p-tmp12reu.bank05.asm` (`cmd_memcpy`, right after `cmd_cp`), no server-side changes needed. Deliberately reuses the exact technique BASIC's own `SAVE`/`LOAD` tokens use internally (`docs/inspiration/c64/mapc6411.txt`'s `$E156`/`$E168` writeups): for `SAVE` ($FFD8), the zero-page pointer argument is satisfied by temporarily repointing `TXTTAB` (`$2b/$2c`, BASIC's start-of-program pointer) at the requested start address, restored immediately after regardless of success; the end address is passed directly in X/Y (as `$end+1`, since KERNAL `SAVE` wants "address of the byte after the last byte to save" but the user-facing syntax treats `$end` as inclusive). For `LOAD` ($FFD5), no zero-page trick is needed at all ‚Äî X/Y holds the start address directly, and `SETLFS`'s secondary address is set to **0** (rather than the `1` a plain `LOAD"name",8,1` would use), which is exactly what tells the KERNAL LOAD routine to ignore any load address embedded in the file and use X/Y instead ‚Äî this single flag is what makes the restore relocatable. No new file I/O was written: this is a straight `SETLFS`/`SETNAM`/`LOAD`/`SAVE` call, so it rides bank01's pre-existing `ILOAD`/`ISAVE` fast-path patch to Ultimate DOS the same way a plain `SAVE"name",8,1` typed at the BASIC prompt already would.
- **Device resolution:** `H`/`T`/`F`/`8` all resolve to KERNAL device number 8 (they're the same physical Ultimate drive; `#h`/`#t`/`#f` only changes its *current directory* via UCI DOS `CHANGE_DIR`, same as `cd`), `9`‚Üí9, `S`‚Üí10 (SoftIEC). The filename argument may optionally be prefixed `#X:` (e.g. `memcpy $c000-$cfff #h:dump.bin`) to override the device **for that one command only** ‚Äî `w_dev` itself is left untouched, unlike `#c`/`#n`'s persistent switch. `C`/`N` have no local-memory bridge (there's nothing on the server side to receive/send a raw memory range against) and report `NOT SUPPORTED`, matching `cp`'s existing device-scoping convention.

New layout, chosen so that **no LOADable program address ever overlaps wedge state** (normal PRGs load at `$0801+`; even `$0400` loads don't reach pages 0-3): `$0110-$014F` = `wc_buf` (`$0100-$010F` deliberately skipped: BASIC's FOUT/LINPRT builds number strings there, and `ll` prints block counts through LINPRT); `$02A7-$02F0` = `cf_shadow` (73 chars + null; the freeze button's resume path scribbles `$02A7-$02CA` ‚Äî harmless, it's rewritten every line); `$02F1-$02FB` = overlay union of the net scratch / memcpy scratch / reset stub (never live simultaneously), `$02FC-$02FF` = `w_dig*`/`w_hidx`/`w_hstart` (live concurrently with the net set); `$0340-$03E6` = the resident stubs in the datassette buffer (IERROR stub + xb2 at `$0340-$0370`, ILOAD shim at `$0372-$039D`, CINV stub at `$03A0-$03E6`), placed past the freezer's `$0334/$0335` resume vector and the ML monitor's `$0336-$033B` state; `$03E7-$03FB` = the state block (`ie_orig_vec`‚Ä¶`w_magic`), ending exactly on the last datassette-buffer byte. Both `.asm` files carry the duplicated `.const` blocks ‚Äî keep them in sync ‚Äî and both now have **`.errorif` layout guards** so a template growing past its slot fails the build instead of silently corrupting the neighbor at runtime (this guard caught a real 2-byte overlap during this round: the CINV stub is 71 bytes, not the 69 the old comment claimed).
- **Refresh-every-line**: `wedge_install` no longer skips the stub copies when the vectors look installed ‚Äî the CINV stub and ILOAD shim bytes are re-copied from ROM templates on every typed line, so any corruption (a monitor session, freeze/resume, a program that scribbled the tape buffer) heals on the next line. One-time state defaults (`w_dev` etc.) are now keyed on a 2-byte magic (`w_magic` = `$A5,$C3`) instead of the vector check, so the current device is **not** reset to `H` by every disarm/rearm cycle.
- **Investigated and rejected: hosting the stubs in the cart's `$DE00-$DEFF` page.** That page is not RAM ‚Äî it's a **per-bank ROM mirror of each bank's `$9E00-$9EFF`** (that's how the cross-bank trampolines work: every bank carries identical bytes at the shared offsets, so `sta $de00` mid-routine lands the next fetch in the new bank's identical mirror). Truly load-proof and freeze-proof, but a byte-scan across all 8 banks found only 37 common-free bytes, fragmented (largest run: `$DE02-$DE0F`, 14 bytes ‚Äî which is exactly the RR **clock-port window**, the likely reason the firmware leaves it empty), and a bank-switch gate needs ~9-11 contiguous common bytes. Not viable without risking clock-port conflicts; documented here so it isn't re-derived.
- **Residual exposure, documented deliberately**: (a) a program whose *file* loads into `$0334-$03FB` (rare tape-buffer stager PRGs) loses those bytes if a line is typed before launching it ‚Äî everything else, including all normal `$0801+` loads, is untouched; (b) saving to *tape* while armed would let the KERNAL write headers over the armed stubs (loads are covered by the shim; C64U tape use is essentially nil); (c) entering a program without LOAD/RUN/SYS (e.g. a future `g`-style jump) runs with hooks armed ‚Äî if a `g`/`run` command is added later, it must call `disarm_hooks` first; (d) the ML monitor may trample the state block (`w_dev` resets gracefully via the magic check).
- **Build/verification (2026-07-04, round 6):** builds clean; diff-localization: **4185 bytes differ, banks 01+05 only** ‚Äî bank01 `$83FF-$8401` hook + `$9D74-$9E00` pocket (135 bytes: line_tap reorder + `w_skip` check + the 3-byte bit7 fix; 15 pocket bytes still free), bank05 `$8023-$901A` (module + jump table targets); bank01 `$9E10+`, bank05 `$9200-$9DFF` and `$9E00+` byte-identical to stock; stub templates byte-verified in the built image (txa/bmi entry, shim's vector restores, jsr `$9012`/`$9015` cross-bank calls). **Round 3 hardware feedback (2026-07-04, fourth pass) and what changed:**
- **First typed line after a fresh boot silently vanished** (no output, no `READY.`, just a blinking cursor ‚Äî retyping the exact same line then worked). Root-caused via careful re-reading of `line_tap` (bank01) against the code it wedges into: `bank01_sub_8407` (the cart's stock scanner, unmodified) branches on the **carry flag** immediately on entry (`bcs` = non-digit first char ‚Üí keyword scan / direct execution, falls through = digit first char ‚Üí program-line entry), and that carry has to be whatever `CHRGET` left it as. `line_tap` was restoring that saved carry (via `plp`) right after its own `bne`/empty-line check, then running the shadow-copy loop and the IERROR-install idempotency check *afterward* ‚Äî both of which contain `CPY`/`CMP` instructions that silently clobber carry before the final `jmp bank01_sub_8407`. On a cold boot, the one-time `lt_install` path's last carry-affecting comparison happens to leave carry in whatever state the *stock* IERROR vector's high byte compares to `$CF` ‚Äî an incidental, wrong value; on every later line, the steady-state `lt_installed` fast path always ends its own comparison with carry set (an equal-value check), which is *also* not the real CHRGET-derived value but happens to be right for typical (non-digit-led) input, which is why retyping the same line always "fixed" it. Fixed by deferring the `plp` to immediately before `jmp bank01_sub_8407` ‚Äî the pushed flags now ride the stack, untouched, through the entire shadow-copy/install/CINV-refresh sequence, so the scanner always sees the real `CHRGET` carry regardless of what our own code compared in between. This also fixes a second, latent bug the old code had: actually typing a *program line* (e.g. `10 PRINT "HI"`) would have hit the same steady-state "always carry set" problem and been misrouted every time, not just on the first line. **Not yet hardware-retested.**

While on `#h`/`#t`/`#f`, `cp <name>` is client-side rewritten to `"PUT <name>"` (stash the argument in `wc_buf`, rebuild `cf_shadow`) before forwarding; the server FTP-*pulls* (`retrbinary`) the file back out of the C64U's `/temp` into the `#n` session's current directory. **Deliberately, explicitly scoped this way for the first pass**: the local side of any `cp` is always anchored on `/TEMP`, never "whatever `cd` took you to" ‚Äî the wire protocol has no round trip to query the Ultimate DOS current directory from the server side, so relaying an arbitrary local path isn't possible without adding one, and this wasn't attempted yet. `8`/`9`/`S`/`C` report `NOT SUPPORTED` for `cp` (no local filesystem / no `#n` session to bridge). **None of this is hardware-tested yet** ‚Äî the earlier back-and-forth in this same round explored (and the user explicitly ruled out) two other designs before landing here: pointing the C64 itself at a literal FTP server as a full 6502 FTP client (rejected: enormous new protocol surface, redundant with `H`/`T`/`F` when the target would've been the Ultimate's own server), and the C64 connecting outward to the Ultimate's *own* built-in FTP server from itself (also rejected as redundant with `H`/`T`/`F`). The server-proxies-via-FTP-to-the-client design above reuses code that's already running in production (CSDB's file downloads) rather than inventing anything new for the transport.

## 1. What this project is

`rr38p-tmp12reu.bin` is a 65536-byte, 8-bank (8192 bytes/bank) ROM image for a Retro Replay cartridge, running a custom "CyberPunX" firmware build (freeze menu, machine-code monitor, flash utility, Turbo Macro Pro assembler+REU, and a BASIC toolkit with a resident command wedge). The goal of this project is to add custom commands to that resident wedge ‚Äî starting with `HONDANI` (increments border color, as a proof of concept) and going forward, a fuller set of commands ("hondani shell") that should invoke logic to be developed in a separate project/repository.

### Files in this set
- `rr38p-tmp12reu.bin` ‚Äî the **original, unmodified** stock ROM dump. Kept as a reference/diff baseline ‚Äî do not edit.
- `rr38p-tmp12reu.bank00.asm` ‚Ä¶ `bank07.asm` ‚Äî Kick Assembler source for each 8KB bank, disassembled from the stock ROM. **These are the live source of truth** ‚Äî edit these, then rebuild.

## 2. How the cartridge's own wedge works (confirmed by tracing, not guessed)

There is **no classic IGONE-vector wedge** (`$0308/$0309`, the textbook 1541-DOS-Wedge mechanism) in this firmware. Instead, bank01 implements a **generic keyword scanner hooked through IMAIN**:

- Bank01's cold start patches `$0302/$0303` (**IMAIN**, BASIC's main-loop soft vector) to `$DE8F` ‚Äî a trampoline in the cartridge's I/O2-mapped RAM. Every time BASIC returns to its main loop (i.e. after every submitted line), execution routes through `$DE8F` into `bank01_api_02` (ROM address `$83D0`).
- `bank01_api_02` gates on a flag at `$0313`, then (if set) calls down into `bank01_sub_83e9`, which fetches the current character via CHRGET (`$0073`) and, for a non-empty line, falls into the scanner at `bank01_sub_8407` (ROM `$8407`).
- The scanner walks a keyword table at `$8507`, comparing typed input (via `$7A/$7B` TXTPTR, indirect-indexed) against each stored word. Words are PETSCII with **bit 7 set on the last character** as an end-of-word marker (same convention BASIC ROM's own tokenizer uses). On full match, `word_index * 2` indexes a parallel jump table at `$8572` to find the handler address.
- **Abbreviation quirk:** once the first 3 characters of a word match, the scanner stops requiring an exact match and just swallows further letters (`>= 'A'`) unconditionally. E.g. the stored word `TASS` also fires for `TASX`/`TASM`/etc. ‚Äî this is how typing `TASM` launches Turbo Macro Pro (canonical command is `TASS`).

## 3. Why we integrate into bank01 instead of hooking our own vector

**Tried and empirically ruled out:** installing an independent hook on IGONE (`$0308/$0309`) from a standalone program. Confirmed via live memory dumps (VICE monitor) that bank01's own housekeeping resets `$0308/$0309` back to the KERNAL/BASIC stock default (`$A7E4`) on **every single line processed**, including before any user input is even evaluated. An independently-installed IGONE hook cannot survive on this cartridge ‚Äî it gets wiped before it's ever useful. Do not retry this approach.

The only mechanism that reliably survives is being part of bank01's own scan path itself (since that *is* the thing doing the resetting). This requires editing bank01's ROM source directly and rebuilding ‚Äî which is exactly what `build.sh` + Kick Assembler lets us do properly.

## 4. Working technique: adding a command via ROM source edit

### The constraint that shapes everything

Bank01's disassembly uses **raw hardcoded hex addresses** for indexed table lookups (e.g. `lda $8507,x`, `lda $8572,x` ‚Äî not symbolic labels), even though branches/jumps elsewhere mostly use symbols. This means: **never change the total byte length of anything before these references, or any code between them and the file's end** ‚Äî doing so silently invalidates every hardcoded address after the insertion point, across a ~2500-line file that hasn't been fully audited for every such reference. Two safe patterns only:

1. **Same-byte-count instruction swaps** ‚Äî replace N bytes of existing code with a different N bytes (e.g. a 3-byte `tax`/`bne` pair with a 3-byte `jmp`). Nothing shifts.
2. **Fill confirmed-zero padding regions** with new code, keeping the *total byte count of the replaced block identical* (pad the remainder with `.byte $00` rows to match). Locate real free space via a raw byte scan of the `.bin` for long runs of `$00` (or `$20`/`$A4`/etc. in other banks) ‚Äî **not** by trusting "reachable bytes" from a naive disassembler pass, which undercounts free space in banks with complex control flow (e.g. banks 6/7's Turbo Macro Pro).

### The HONDANI implementation (done, built, byte-diff-verified)

1. **Hook point:** `bank01_sub_83e9` (part of the path from `bank01_api_02` ‚Üí scanner), specifically the 3 bytes `tax` / `bne bank01_sub_8407` at ROM `$83FD`-ish (right after `jsr $0073` / CHRGET). Replaced with `jmp hondani_check` ‚Äî also exactly 3 bytes. This runs **before** the cartridge's own table scan (`bank01_sub_8407`) even starts.
2. **Free space used:** the confirmed 156-byte zero-padding run at `$9D74`-`$9E0F` in bank01 (found via raw `.bin` byte scan, not disassembler reachability). `hondani_check` occupies the first 64 bytes (`$9D74`-`$9DB3`); the `HONDANI` string data (`hword`) sits at `$9DAD`-`$9DB3`; **92 bytes remain free** at `$9DB4`-`$9E0F` for further commands.
3. **Correctness requirement:** `hondani_check` must exactly replicate the original `tax`/`bne bank01_sub_8407` semantics for every input that ISN'T our new command, since bank01_sub_8407 depends on the **carry flag** left by CHRGET (tests it immediately via `bcs`) and on `X` holding the typed character (for the empty-line case, `stx $0258`). Solved with `php`/`plp` bracketing around our own comparison logic, so flags are restored byte-for-byte before falling through to the untouched original scanner on any non-match. This is the pattern to replicate for additional commands ‚Äî see ¬ß6.
4. **Return path on match:** advance TXTPTR past the matched word, then `jmp ($0302)` (indirect through IMAIN) to cleanly re-enter BASIC's main loop. This is deliberately different from the existing table's own return convention (`bank01_sub_848d` uses an elaborate RTS-trick through `$0073`/CHRGET) ‚Äî our injected code isn't part of that call-depth, so the simpler IMAIN re-entry is correct and safe.

Exact addresses in the current build (`build/rr38p-tmp12reu.rebuilt.bin`), verified by direct byte search, not estimated:
- `jmp hondani_check` opcode: ROM `$83FF` (bank01)
- `hondani_check` entry: ROM `$9D74` (bank01)
- `hword` (`"HONDANI"` bytes): ROM `$9DAD` (bank01)

Full current source of the routine lives in `rr38p-tmp12reu.bank01.asm` at the `hondani_check:` label (search for it) ‚Äî read that directly rather than relying on this summary if reimplementing/extending.

**Update:** `hmatched`'s body no longer does `inc $d020` inline ‚Äî it now relocates a small stub (`xb_stub`) to RAM and calls out to bank05 to do it. See ¬ß6a for why and how. This grew the pocket's used footprint from 64 to 93 bytes, leaving **63 bytes free** at the (now later) end of the padding run, still ending at `$9E0F`.

## 6. How to add more commands (the "hondani shell" path forward)

Don't hunt for a new hook point per command ‚Äî reuse the one already wired in (`hondani_check`, entered before the cartridge's own scanner). Generalize it into a small dispatcher:

1. After the existing `HONDANI` check's `nothondani` fails, instead of immediately falling through to `plp`/`jmp bank01_sub_8407`, add another compare block for the next keyword before that fallback ‚Äî same pattern (`ldy #0` / compare loop / boundary check / branch to its own `matched` label). Keep the **final** fallback (`plp` + `jmp bank01_sub_8407`) as the last thing tried, after all custom keywords have had a chance.
2. Each new "matched" handler can do its own thing directly, or ‚Äî more likely useful for a "shell" ‚Äî call out to logic living in bank05 via the `xb_stub` mechanism (¬ß6a). Keep growing small dispatcher logic itself in bank01's pocket (mind the byte budget ‚Äî check with the free-space scan technique above, don't just assume); put the actual command *bodies* in bank05, which now has ~1200+ bytes of confirmed-free space across several pockets (see ¬ß6a) since its stock flash-utility role isn't needed anymore.
3. **Every time**, before flashing: rebuild, run the diff-localization check (¬ß5), and confirm the touched byte count/region matches what you intended. This is the guardrail against silently breaking the raw-hex-address references described in ¬ß4.
4. Keep the abbreviation-matching convention in mind if you want it (3-letters-then-swallow-rest, matching the cartridge's own style) ‚Äî our current `HONDANI` check requires an exact 7-letter match plus a non-letter boundary; it does not implement the abbreviation quirk. Decide per-command whether that's desired.

## 6a. Cross-bank calling: the `xb_stub` mechanism (bank01 ‚Üí bank05)

**Why this exists:** `hondani_check` runs while bank01 is paged into `$8000-$9FFF`. To run code that lives in bank05 instead, something has to write bank05's select value to `$DE00` ‚Äî but the instruction that does that write, and everything after it up to the point where we jump into bank05, is *itself* sitting in the very `$8000-$9FFF` window about to be repaged. If that switching code lived in bank01's ROM at, say, `$9D90`, the CPU would fetch the `STA $DE00` fine, but the *next* instruction fetch (still at an address in `$8000-$9FFF`) would now come from bank05's ROM instead of bank01's ‚Äî the code silently falls into whatever garbage bytes bank05 has at that same address. (This is exactly why the stock firmware's own freeze/NMI handler code near `$9EB0-$9EF0` is byte-identical across every bank ‚Äî same trick, solved by making the switching code identical wherever it lands.)

Our solution: don't solve it by duplication ‚Äî relocate the switching code to real C64 RAM at **`$C000-$CFFF`** (specifically `$CF00`), which is untouched by any `$8000-$9FFF`/`$A000-$BFFF` cartridge banking in any standard C64 memory configuration. `hondani_check`'s `hmatched` path copies an 18-byte relocatable stub (`xb_stub`, defined right after `hword` in bank01's pocket) to `$CF00` on every match (cheap, idempotent, no install-once bookkeeping needed), then `jsr $cf00`. The stub itself:
```
sei
pha
lda #$88        ; bank05 select value
sta $de00
jsr $9012       ; bank05 entry point
lda #$08        ; bank01 select value -- restore
sta $de00
pla
cli
rts
```
Because this code runs from `$CF00` (never repaged), the `jsr $9012` executes correctly the instant `$8000-$9FFF` shows bank05's ROM, and when bank05's routine does `rts`, control returns to the still-resident `$CF00` code to switch back. `xb_stub` has no internal branches/relative addressing, so it's safe to relocate byte-for-byte.

**The `$DE00` bank-select bit layout ‚Äî confirmed against official hardware documentation.** Every one of the 8 banks embeds its own "which bank am I" marker byte in its copy of the shared freeze/NMI trampoline footer (`$9EED` in the disassembly). Reading all 8 gave a self-consistent decode which was later checked against the VICE/hardware reference for cartridge type 36 ("Retro Replay") and matches it exactly ‚Äî see ¬ß7 for the full register documentation and the cross-check. Summary:
| Bank | Marker | Binary | Bank number encoding |
|---|---|---|---|
| 0 | `$00` | `00000000` | bit3=0,bit4=0,bit7=0 |
| 1 | `$08` | `00001000` | bit3=1 |
| 2 | `$10` | `00010000` | bit4=1 |
| 3 | `$18` | `00011000` | bit3=1,bit4=1 |
| 4 | `$80` | `10000000` | bit7=1 |
| 5 | `$88` | `10001000` | bit7=1,bit3=1 |
| 6 | `$90` | `10010000` | bit7=1,bit4=1 |
| 7 | `$98` | `10011000` | bit7=1,bit4=1,bit3=1 |

i.e. **bank number = bit3 (LSB) + bit4√ó2 + bit7√ó4**. Per ¬ß7, these are documented as "bank-address 13/14/15" ‚Äî exactly the three bits that select an 8KiB (`2^13`-byte) bank within the cart's ROM/RAM. Bits 0/1 (GAME/EXROM) are left `0` in every marker, which per ¬ß7 means GAME deasserted + EXROM asserted ‚Äî the standard "8K game" mapping (ROM only at `$8000-$9FFF`), matching this cart's own CRT header defaults. Bit 2 (write-disable/decartify) and bit 5 (ROM-vs-RAM select) are `0` ‚Äî cart stays enabled, ROM (not RAM) is what's being banked. Bit 6 (freeze-exit latch) is irrelevant outside of exiting a freeze. All of this is now **confirmed by the vendor/VICE documentation, not just cross-referenced ROM inference.**

**Bank05 side:** `hondani_shell_entry` (`inc $d020` / `rts`, 4 bytes) lives at ROM `$9012` in bank05, inside a confirmed-zero 494-byte free run at `$9012-$91FF` (found via the same raw-byte-scan technique as ¬ß4, freshly re-scanned ‚Äî not the same numbers as an earlier, less precise pass). Bank05's cold-start vector table, jump table, and all other existing code are **untouched** ‚Äî this is a minimal, reversible edit, not a wholesale bank takeover. *(Historical note ‚Äî superseded: as of ¬ß11 the flash-utility reserve at `$8023-$9011` **has** been taken over for the shell module, `$9012` is a fixed jump table, and the current free-space picture is described there.)*

**Extending this for more commands:** the `xb_stub` template hardcodes both the bank05 entry address (`$9012`) and the "return to bank01" value (`$08`) ‚Äî it's a fixed single-purpose call, not yet a generic "call any address in any bank" primitive. Adding a second bank05-backed command means either (a) adding a second small fixed stub the same way, or (b) generalizing `xb_stub` to accept a target address (e.g. via a zero-page vector at `$FB`/`$FC` and an indirect `jmp ($00fb)`, since self-modifying the stub's own operand bytes won't work once it's sitting in RAM ‚Äî actually it *would* work there since `$CF00` is RAM, unlike the ROM template; either approach is viable). Worth doing once there are 2+ bank05 commands to justify it.

## 7. Reference: official Retro Replay (`$DE00`/`$DE01`) register documentation

This is the authoritative hardware reference (VICE cartridge type 36, "Retro Replay") that confirms the bank-select bit layout derived empirically in ¬ß6a. Kept here verbatim (condensed) so this file stays self-contained.

Cartridge config for this ROM: 64KiB (8 banks of 8KiB), **EXROM active/lo (0)**, **GAME inactive/hi (1)**, load address `$8000-$9FFF` ‚Äî i.e. standard "8K game" mapping, ROM only at `$8000-$9FFF`.

Two write-only registers (`$DE00`, `$DE01`); reading either returns a third, separate status register. `$DE00` resets to `$00` on a hard reset (unless flash-mode jumper is set, then `$02`).

**`$DE00` (write):**
| bit | meaning |
|---|---|
| 0 | GAME line: 1=assert, 0=deassert |
| 1 | EXROM line: 0=assert, 1=deassert |
| 2 | 1 = disable further register writes, force standard C64 memory map (as if no cart installed) |
| 3 | bank-address 13 (ROM/RAM banking) |
| 4 | bank-address 14 (ROM/RAM banking) |
| 5 | 0=ROM, 1=RAM |
| 6 | write 1 once after a successful freeze to restore normal memory map + re-enable bits 0/1; no effect otherwise |
| 7 | bank-address 15 (ROM banking) |

**`$DE01` (write, extended control register):** if not in flash mode, bits 1/2/6 are write-once.
| bit | meaning |
|---|---|
| 0 | enable clockport connector |
| 1 | AllowBank ‚Äî 1 allows RAM banking in the `$DF00`/`$DE02` mirror area |
| 2 | NoFreeze ‚Äî 1 disables the freeze button |
| 3 | bank-address 13 (mirror of `$DE00` bit 3) |
| 4 | bank-address 14 (mirror of `$DE00` bit 4) |
| 5 | bank-address 16 (ROM, flash mode only) |
| 6 | REUcomp ‚Äî 0=standard memory map, 1=REU-compatible memory map |
| 7 | bank-address 15 (mirror of `$DE00` bit 7) |

**Status register (read from either `$DE00` or `$DE01`):**
| bit | meaning |
|---|---|
| 0 | 1=flash-mode jumper set |
| 1 | feedback of AllowBank |
| 2 | 1=freeze button pressed |
| 3 | feedback of banking bit 13 |
| 4 | feedback of banking bit 14 |
| 5 | feedback of banking bit 16 |
| 6 | 1=REU-compatible memory map active |
| 7 | feedback of banking bit 15 |

**Memory maps:**
- **Standard** (what this project uses) ‚Äî `$DE00`/`$DE01` active; `$DF00-$DFFF` mirrors the last page of the selected bank (ROM or RAM per bit 5); RAM accessible only at `$8000-$9FFF`; ROM maps to `$8000`, `$A000`, or `$E000` depending on GAME/EXROM. If AllowBank is unset, `$DF00-$DFFF` always shows RAM bank 0 (back-compat with older cart images); doesn't affect the ROM mirror there.
- **Freeze** ‚Äî ROM forced to `$E000-$FFFF`, bank 0, right after freeze; bits 0/1 of `$DE00` have no effect on GAME/EXROM while frozen; should be exited ASAP (write bit 6) since C64 RAM at `$E000` isn't available and GAME/EXROM aren't under program control in this mode.
- **REU-compatible** ‚Äî same as standard, except `$DE02-$DEFF` mirrors `$9E02-$9EFF` of the selected bank instead of `$DF00-$DFFF`, leaving `$DF00` free for a real 1764 REU.

**Cross-check against this project's usage (¬ß6a):** `xb_stub` writes only bits 3/4/7 (bank-address bits) with bits 0/1/2/5/6 always `0`. Per this table that means: GAME deasserted + EXROM asserted (standard 8K game map, matches the CRT header), registers stay enabled (bit 2=0), ROM selected not RAM (bit 5=0), no freeze-related effect (bit 6=0, not in a freeze). This is exactly the "just switch the ROM bank, leave everything else in its normal running state" operation `xb_stub` is meant to perform ‚Äî confirmed correct, no implementation changes needed. Also confirms `$C000-$CFFF` (where `xb_stub` relocates itself to survive the bank switch) is never touched by any of the three memory maps above, so it's safe under all of them, not just the one state that happened to be inspected empirically.

## 8. Chatbot fallback command (in progress)

**Goal:** port the "unrecognized command ‚Üí send to HDN Cloud" behavior from the standalone `hdnsh` project (`src/cmd_unknown.asm`, `src/hdncloud.asm`, `src/c64u_network.asm`, `src/c64u_common.asm` in the repo root, *not* this `wedge/` directory) into the wedge cartridge. User's own natural-language messages, typed at the BASIC prompt, should be forwarded to the same cloud chatbot backend (`cloud/` in the repo root) as free text.

**Key difference from the old `hdnsh` context:** the original code was flashed as a **replacement BASIC ROM** (`Makefile`'s `run-c64u` target FTPs `hdnsh.bin` to `Flash/roms/` and reboots; `run-hdn-vice` uses VICE's `-basic` flag) ‚Äî it owned the whole machine and had its own command parser, so "unrecognized" simply meant "didn't match any entry in my own command table." Here, stock C64 BASIC is still fully alive underneath the wedge, so "unrecognized" has to mean something more specific: a line that's neither valid BASIC nor one of our own wedge keywords. **Decision (user, 2026-07-03): implement this as a true fallback**, not a new dedicated keyword ‚Äî i.e., eventually hook the point where BASIC itself would otherwise print `?SYNTAX ERROR`, not another entry in `hondani_check`'s pre-scan table. That hook point hasn't been built yet (see open items below); bank01 doesn't touch the standard `IERROR` vector (`$0300/$0301`) anywhere in its own housekeeping, unlike `IGONE` (¬ß3), so it's a promising candidate ‚Äî untested so far.

**Decision (user): reuse the existing wire protocol as-is.** Same `HDNCLD_CMD_*` opcodes, same `$FE` packet magic byte, same TCP port 6464, same `cloud/` Python server ‚Äî no server-side changes planned. The C64 side talks to the cloud over TCP via the **Ultimate Command Interface** at `$DF1C-$DF1F` (`CONTROL_REG`/`STATUS_REG`=`$DF1C`, `CMD_DATA_REG`=`$DF1D`, `RESP_DATA_REG`=`$DF1E`, `STATUS_DATA_REG`=`$DF1F`) ‚Äî a documented Ultimate64/C64U hardware feature, unrelated to and separate from this cart's own `$DE00`/`$DE01` registers (¬ß7).

**The one big open risk, flagged by the user as unverified: does `$DF1C-$DF1F` work correctly while this RR-style cart is actively banking?** The old `hdnsh` never had to find out, since it replaced BASIC ROM outright and had no competing cartridge claiming `$DF00-$DFFF`. Our wedge cart, per ¬ß7, claims that entire address range for its own I/O3 mirror by default (`$DF00-$DFFF` shows the last page of the selected ROM/RAM bank). Whether Ultimate's FPGA gives its own command interface priority at those 3 specific addresses regardless of cartridge I/O3 claims is *believed* likely (Ultimate's own source comment in `src/c64u_common.asm` calls these "Ultimate cartridge registers," implying they're meant to be reachable from within cart contexts) but has **not been confirmed** on this specific hardware/firmware combination. This is the reason for the minimal test below, rather than porting the whole fallback mechanism sight-unseen.

**RESOLVED (user tested on hardware, 2026-07-03): `$DF1C-$DF1F` works correctly while bank05 is paged in via the wedge's own bank-switch.** Typing `HONDANI` flashed the border green and printed `?ERROR` on screen; the cloud server log showed `Received: 'ping'` followed by `No handler found for the request.`. This is the *correct* server response, not a failure ‚Äî `request_dispatcher.py` only routes text starting with `"I:"` to `ChatHandler` (see `cloud/handlers/chat_handler.py`'s `can_handle`); anything else falls through every handler and gets the literal string `"?ERROR"` (`cloud/request_dispatcher.py`, end of `dispatch()`). Since the connect ‚Üí write ‚Üí server-received-and-logged-it ‚Üí dispatch ‚Üí response ‚Üí C64-read-and-printed chain completed end-to-end while bank05 was actively paged in, the hardware risk this whole test existed to check is now closed. **New fact this surfaced for Step 2:** to actually reach the chat/LLM handler, the forwarded text needs to be prefixed with `"I:"` ‚Äî that's the server's existing routing convention, and the C64 side will need to prepend it (see Step 2 item 2 below), not just forward the raw typed line.

### Step 1 (done, confirmed on hardware 2026-07-03): minimal cloud round-trip test

Rather than build the full IERROR-hook fallback blind, `bank05`'s `hondani_shell_entry` (still reached the exact same way as the already-confirmed-working `HONDANI` command, via bank01's `xb_stub` ‚Äî see ¬ß6a) was rewritten from `inc $d020` into a **self-contained cloud connectivity test**, isolating the one genuinely unverified variable (`$DF1C` reachability) from everything else:
- Connects to the cloud server at a **hardcoded test address (`192.168.1.2:6464`, placeholder ‚Äî confirm/update `net_test_host` in `rr38p-tmp12reu.bank05.asm` if your `cloud_server.py` runs elsewhere)** via `NET_CMD_TCP_SOCKET_CONNECT`.
- Sends a fixed `TEXT_INPUT` ($02) packet for console 0 containing the literal text `"PING"`.
- Streams whatever comes back straight to the screen via `CHROUT` ($FFD2) ‚Äî even a "processing error" reply from the LLM dispatcher counts as success here, since the point is proving the wire works, not exercising the chatbot itself.
- Closes the socket.
- Flashes the border **green ($05)** if every protocol step (connect/write/read/close) completed without the Ultimate interface reporting an error, **red ($02)** if any step failed at the protocol level.

Implementation notes:
- Uses **no zero page at all** ‚Äî all scratch state (`socket_id`, two status bytes) lives in RAM at `$CF20-$CF22`, right after `xb_stub`'s own `$CF00-$CF11`, both safely outside any bank-switched window (¬ß6a). This sidesteps a real open question (see below) about whether the old `hdnsh`'s zero-page choices ($9B/$9C, $C1/$C2, etc. ‚Äî see `src/constants.asm`) are actually safe to reuse while stock BASIC/KERNAL are alive; that question is deferred, not answered, by avoiding zero page entirely for this test.
- The protocol logic (`net_wait_idle`, `net_wait_not_busy`, `net_push_and_check`, `net_drain_data`, `net_print_data`, `net_read_status`, `net_accept`) is a fairly literal, self-contained port of `src/c64u_common.asm`'s `wait_idle`/`wait_not_busy`/`uii_readdata`/`uii_readstatus`/`uii_accept`/`uii_success`, with the dead `tay`/`tya` round-trip in the original `wait_not_busy` dropped and the callback-pointer indirection (`JSR_INDIRECT_ADDR`) replaced by a direct `jsr $ffd2` since this test always wants CHROUT.
- Lives entirely in the same confirmed-free 494-byte pocket at `$9012-$91FF` in bank05 that `hondani_shell_entry` already occupied. Verified via the same diff-localization technique as before after every change: `$9200` onward in bank05 stays byte-identical to stock ‚Äî nothing shifted.
- **Known rough edge, not yet addressed:** `xb_stub` wraps the entire call to bank05 in `sei`/`cli` (¬ß6a), so interrupts are disabled for the *entire* network round trip, however long that takes. Harmless to correctness (nothing here depends on IRQ timing) but the machine will appear frozen ‚Äî no blinking cursor, no keyboard buffer updates ‚Äî for the duration. Fine for a one-shot hardware test; worth revisiting if/when this becomes the real fallback path, since a slow/unresponsive cloud server would freeze the machine for that whole time.

**Bug #1 found and fixed (2026-07-03) after the first real-chat test:** with `"PING"` (which just got the short literal `"?ERROR"` back), the naive single-shot read worked. Switching the test message to `"I:HELLO"` to trigger a real LLM reply (~70+ characters, multi-line) surfaced a real protocol bug: only **two garbage-looking characters (`$5E`)** printed instead of the actual reply. Root cause: a single `SOCKET_READ` command only drains whatever's queued *at that instant* ‚Äî it does not wait for or guarantee the full response. The original `src/hdncloud.asm`'s `send_cmd_to_hdn_cloud` handles this with a loop (`!repeat_until_00data:` ... `jsr uii_socketread` ... `bcs !repeat_until_00data-`) that **reissues `SOCKET_READ` until the status queue reports `"00"` (fully drained)** ‚Äî our first cut of `hondani_shell_entry` skipped that loop entirely and only ever issued one read, so it silently only saw the first (tiny, partial) chunk. Fixed by wrapping the read/print/status/accept sequence in the same kind of loop.

**Bug #2 found and fixed (2026-07-03), same test rerun after bug #1's fix:** border still green (protocol-level success), but now **40 repeated `$5E` characters** and no real text ‚Äî worse-looking, but actually a more precise clue. `$5E` is `KEY_ARROW_UP` (`src/constants.asm`), and that file also documents (but `src/c64u_common.asm`'s actual `uii_readdata_network` never implements!) a 3-state response framing: `SOCKET_WRITTEN` ‚Üí first byte is a marker (documented as expecting `$13`), `SOCKET_AFTER_13` ‚Üí "ignore all `$5e` bytes until next other byte", `SOCKET_READING` ‚Üí real content follows. Implemented that 3-state machine (`SOCKET_WRITTEN`/`SOCKET_AFTER_13`/`SOCKET_READING` in `cf_socket_status`).

**Bug #3 found (2026-07-03), rerun after bug #2's fix:** partial progress ‚Äî real content **did** print this time (`Hello Honza! ... What would ... today?`), proving bug #1's retry loop is doing its job. But it was still preceded by **41 unfiltered `$5E` characters and a stray leading `:`** ‚Äî the 3-state swallow logic did not swallow anything, despite the code looking correct on review (traced through every branch by hand, found nothing wrong on paper). Rather than keep guessing at the exact cause blind (no hardware access to single-step this), **simplified the approach to be more robust regardless of the precise mechanism**: instead of tracking 3 distinct states, `net_read_and_print` now just discards `$13` and `$5E` bytes *anywhere* in the leading run (in any order, any count) via a single `cf_seen_content` flag, and prints unconditionally once the first byte that's neither has been seen. This is a strictly weaker assumption than the 3-state version (doesn't care about exact marker-then-filler ordering) and is simpler code, so less likely to hide the same kind of bug. **Still not hardware-verified** ‚Äî if `$5E`/`:` still leak through on the next test, the next step is byte-level diagnostics (e.g. printing each response byte's hex value instead of the character) rather than another guess at the framing shape.

**Also addressed in this pass, per user request (2026-07-03):**
- **Slow server responses:** the retry budget is now a 16-bit counter (`cf_retries_lo`/`cf_retries_hi`, ~8000 iterations) instead of a single-byte cap of 255, giving much more headroom for a slow LLM reply before giving up.
- **Animated "waiting" indicator:** `net_spin` prints a small 2-frame blinking indicator (`.`/`:`, overwriting itself in place via cursor-left `$9D`).
- `net_spin` (and its 2-byte `spin_chars` table) didn't fit in the `$9012-$91FF` pocket alongside everything else ‚Äî it's relocated to a second, separate confirmed-free pocket at `$9E00-$9E9C` in bank05 (157 bytes, previously untouched).

**Bug #4 found and fixed (2026-07-03), rerun after bug #3's fix:** big step forward ‚Äî the `$5E` filler-discard now worked cleanly (exactly 40, matching between two separate runs), and the real reply text printed correctly after it. **Retested (2026-07-03) after bug #4's fix ‚Äî the spinner worked (a `.` briefly visible before each `$5E`, confirming the busy-wait relocation was right), and filler-discard still worked cleanly (37 `$5E`, varying run to run as expected for network-timing-dependent retry counts). But the one-byte field (this time `'G'`) apparently was NOT discarded ‚Äî described as visibly printed before the real reply text.** Four rounds of fixes based on visual descriptions have each found something real but never fully converged, and by this point it's more efficient to get hard data than to keep guessing:
- The filler-byte count varying every run (40, 41, 40, 37) is expected ‚Äî it's a function of network/LLM timing (more retries needed some runs than others), not a bug.
- The one "stray" byte's value varying (`'3'`, `'W'`, `'G'`) is *consistent* with a length/sequence field as hypothesized ‚Äî but whether it's discarded correctly seems to vary too, which points at something more structural than a simple off-by-one: e.g. the response might be split across more than one data-bearing `SOCKET_READ` chunk in some runs but not others, each with its *own* leading field, which a single "discard once globally" flag (as currently implemented) wouldn't handle correctly.
- Rather than guess a 5th time, **`net_read_and_print` was temporarily replaced with an unfiltered hex dump** (`net_print_hex_byte`/`net_print_hex_nibble`): every raw byte from `UII_RESP` printed as two hex digits plus a space, no interpretation at all. The retry loop, 16-bit budget, and spinner were unchanged.

**Response framing CONFIRMED (2026-07-03) from a real hardware hex-dump screenshot ‚Äî this settles it, no more guessing needed:**
```
[N x $ff idle/filler bytes -- N varies with network/LLM timing]
[2-byte little-endian content length -- e.g. "47 00" = 0x0047 = 71]
[exactly that many content bytes]
```
Verified byte-count-exact: the dump showed `47 00` (length 71) followed by *exactly* 71 bytes that decode to `"Hello Honza!\r\rHow can I help you with your C64 and Hondani Shell today?"` ‚Äî capital letters as PETSCII `$C1-$DA` (high bit set), lowercase as `$41-$5A`, both printable via `CHROUT` with no further translation needed (they displayed as all-caps only because the test screen was in default/uppercase charset mode ‚Äî the bytes themselves are correct mixed-case PETSCII). This also retroactively explains every earlier round's symptoms: the varying filler count was always just timing-dependent retry count, and the "one stray byte" from earlier rounds (`'3'`/`'W'`/`'G'`) was actually the *low* byte of this same 2-byte length field ‚Äî the earlier `$5E`/`$13` marker theory was simply wrong (likely a misreading of an unusual glyph before hex ground truth was available).

`net_read_and_print` now implements this for real: `cf_state` (0=discarding `$ff` filler, first non-`$ff` byte is length-lo; 1=that byte was length-lo, next is length-hi; 2=printing content, counting `cf_msglen_lo`/`cf_msglen_hi` down to zero; 3=content fully printed, discard anything else this response) plus the 16-bit `cf_msglen_lo`/`cf_msglen_hi` countdown. The outer read loop in `hondani_shell_entry` also now exits early once `cf_state` reaches 3 (message fully printed), instead of always waiting for the status queue's `"00"` ‚Äî no reason to keep polling once the known-length content is fully drained. `net_read_and_print` itself didn't fit in the `$9012` pocket once fully built out ‚Äî relocated to the `$9E00` pocket alongside `net_spin`/`net_wait_not_busy_spin`. **Not yet re-tested on hardware** ‚Äî but this is now a confirmed protocol implementation, not another guess, so the expectation bar is "should just work" rather than "let's see what breaks next."

## 9. The real fallback: retiring `HONDANI`, hooking `IERROR` (2026-07-03)

**User decision:** once the cloud round-trip was confirmed clean end-to-end (¬ß8), the `HONDANI` keyword itself was retired ‚Äî it was always just the vehicle for proving out `xb_stub`/networking, not the intended final UX. The wedge now implements the originally-intended behavior: type anything BASIC doesn't recognize, and it gets forwarded to the cloud chatbot automatically.

**Researched first, since guessing wrong here fails silently (a subagent dug through `docs/inspiration/c64/mapc6411.txt`, the most authoritative local source ‚Äî a full annotated Commodore 64 memory map):**
- `IERROR` lives at `$0300`/`$0301`, defaulting to `$A43A` (right after the `JMP ($0300)` at the `ERROR` entry, `$A437`).
- BASIC puts the **error-table index** in `X` before `JMP ($0300)` ‚Äî confirmed both by the local doc and independently by the portable `msbasic` reimplementation's own header comment.
- **The exact index for `SYNTAX` is genuinely uncertain** ‚Äî the local doc's error-table ordering computes to `$0A`, but common disassembly literature elsewhere says `$0B`. Rather than trust either, this build fires on *any* direct-mode error and prints the index it saw, so the real value gets confirmed from actual hardware (same "diagnostic build, then lock in the real value" strategy that resolved the response-framing bugs in ¬ß8).
- **Direct mode detection**: `CURLIN`'s high byte, zero page `$3A`, equals `$FF` in direct/immediate mode. Checking only `$3A` (not also `$39`) is sufficient and standard.
- Bank01 does **not** touch `$0300`/`$0301` anywhere in its own housekeeping (unlike `IGONE`, ¬ß3) ‚Äî but this hasn't been empirically hardware-confirmed to *survive* the way the `IMAIN` patch does, only confirmed by absence-of-reference in the disassembly. Worth watching for.

**Implementation (bank01, `rr38p-tmp12reu.bank01.asm`):**
- `hondani_check` is renamed `line_tap` and completely rewritten. It no longer matches any keyword. On every non-empty typed line it: (1) shadow-copies the raw pre-crunch line text (up to 77 chars) to real RAM at `cf_shadow` (`$CF32`) ‚Äî this has to happen *before* BASIC's own tokenizer runs, which is exactly what this hook point already gave us for free; (2) idempotently installs an `IERROR` hook (checks if already installed, so warm restarts don't double-patch); (3) **always** falls through to the cart's own scanner (`bank01_sub_8407`) exactly as before ‚Äî it's a pure tap now, never intercepts.
- `ierror_stub_rom` is the actual hook logic ‚Äî a small relocatable template, copied once to real RAM at `$CF00` (reusing the address `xb_stub` used to occupy, since `xb_stub`/`hword`/the exact-match logic are all gone). It has to live in RAM, not the `$8000-$9FFF` ROM window, because `IERROR` can fire while *any* cartridge bank happens to be paged in ‚Äî exactly the same reasoning as `xb_stub` (¬ß6a). On entry (`X` = error index): if `$3A` ‚âÝ `$FF` (not direct mode), `jmp` through the saved original vector, `X` untouched ‚Äî normal BASIC error reporting for program-mode bugs is completely unaffected. If direct mode: stash `X` into `ie_errcode` (`$CF82`, diagnostic only for now), switch to bank05, call the (renamed-in-purpose) `hondani_shell_entry`, switch back, then `jmp ($0302)` to re-enter BASIC's main loop cleanly ‚Äî the stock `?SYNTAX ERROR` message never prints for this case.

**Implementation (bank05, `rr38p-tmp12reu.bank05.asm`):**
- `hondani_shell_entry` (still at `$9012`, still reached the identical way) now: (1) prints `ERR=$XX ` first, where `XX` is `ie_errcode` in hex ‚Äî the diagnostic output that will tell us the real `SYNTAX` index; (2) sends `"I:"` + the shadow-copied line (`cf_shadow`) instead of a fixed test string, null-terminated same as before.
- The fixed `net_test_msg` test string is gone (no longer needed); `net_test_host` (the hardcoded server IP placeholder) is unchanged.
- `diag_hexdigit` (the small hex-nibble-print helper) didn't fit in either of the two pockets already in use, so it's relocated to a **third** confirmed-free pocket at `$8023-$80FF` (222 bytes, first use) in bank05.
- **A real bug was caught during this pass, before shipping**: the `.fill` boundary for that new pocket was miscalculated by hand (`$80F6` instead of the correct `$8100`), which silently shifted the entire rest of bank05 by 10 bytes ‚Äî the diff-localization check immediately caught it (a 4334-byte diff instead of the expected ~700), and it was fixed before packaging. This is exactly the failure mode ¬ß4's methodology exists to catch, and it caught it.

**Verified via the same diff-localization discipline as every prior change:** bank01's tail past the edited pocket (`$9E10` onward) and all three of bank05's gaps between its now-three edited pockets (`$8100-$9012`, `$9200-$92D0`, `$9300-$9E00`, `$9E9D`-end) are byte-identical to stock. Total diff: 736 bytes across bank01 (107) and bank05 (629), all inside confirmed-free space.

**Not yet tested on hardware.** Expect: typing a valid BASIC command still works normally; typing garbage prints `ERR=$XX ` followed by a real chatbot reply instead of `?SYNTAX ERROR`; typing something that causes a *different* direct-mode error (e.g. `PRINT 1/0`) will *also* currently get forwarded to chat, since the trigger isn't narrowed to `SYNTAX` yet ‚Äî that's expected for this round, not a bug.

## 12. Console switching (2026-07-04, round 3): C=+CTRL + 1‚Ä¶7

Implements the documented `docs/user_manual/cloud-apps.md` behavior: `C=+CTRL+2‚Ä¶7` switch to the server-side apps (file editor, coding agent, web browser, telegram, RSS, wikipedia ‚Äî consoles 2-7, id = digit<<4), `C=+CTRL+1` returns to the local shell. Works at the BASIC prompt and inside any server console (per user: only those two contexts need to work).

**Round-1 hardware feedback: the trigger was originally plain `C=+digit`.** On real hardware this didn't work usably because Commodore+1..8 is *also* decoded by the stock keyboard-IRQ table-setup routine (`$EB48` in `mapc6411.txt`) as a text-colour change, independent of our own hook ‚Äî we always chain to the original CINV vector afterward (needed for cursor blink / IEC timing / everything else that vector drives), so its default colour-change action fires regardless of what our hook does. Fixed by requiring **SHIFT held too** (`SHFLAG & $03 == $03`).

**Round-2 hardware feedback: `C=+SHIFT+digit` flips the character set.** The caveat flagged at the end of round 1 was confirmed on real hardware ‚Äî every combo press visibly toggled the charset via the same KERNAL routine's separate SHIFT+Commodore special case. Moved the trigger to **C=+CTRL+digit** instead (`SHFLAG & $06 == $06`, checked both in the CINV stub's match and in the modal loop's digit-hop check) ‚Äî CTRL takes decode-table priority in the KERNAL's keyboard setup routine and has no colour-change or charset-toggle special case, so this combo should be free of both prior side effects. - **Detection:** `wedge_install` (bank05 `$9015`, reached via bank01's `xb2` RAM trampoline at `$0361` on **every non-empty typed line** ‚Äî self-healing like the IERROR hook; since round 6 it first sniffs for launch lines and disarms instead, see the round-6 section) copies a 71-byte stub to `cinv_ram` (`$03A0`, datassette buffer since round 6) and points **CINV (`$0314`)** at it. ## 13. PySIC prefix renamed `?` ‚Üí `:`, and wired through the wedge for the first time (2026-07-05)

**Problem (user-identified):** PySIC (`cloud/handlers/python_eval_handler.py`, server-side Python expression evaluator) used a leading `?` as its trigger. But `?` is BASIC's own `PRINT` abbreviation, so any PySIC expression that also happens to be syntactically valid BASIC (e.g. `?2+2`) gets executed locally by BASIC's own `PRINT` and never reaches the IERROR hook at all ‚Äî PySIC only ever fired for the subset of expressions that are invalid BASIC syntax. `:` cannot start a BASIC statement, so it reliably errors and reaches the wedge every time.

**Second problem, found while tracing the fix (pre-existing, independent of the prefix character):** PySIC was never actually reachable from real hardware in the first place. `wedge_dispatch_body`'s catch-all for unrecognized lines is `wd_chat`, which unconditionally prepends `"I:"` before forwarding (¬ß9, ¬ß11). So a typed `?0x2000` reached the server as `"I:?0x2000"`, not `"?0x2000"`. `PythonEvalHandler.can_handle()` only matched a *leading* `?`, so on the wrapped text it never claimed the line at all (handler order in `request_dispatcher.py` ‚Äî Help, PythonEval, Chat ‚Äî didn't matter here); `ChatHandler.can_handle()`, which matches any `"i:"`-prefixed text unconditionally, always claimed it instead, so the line was swallowed as a chat message. Same class of bug as the `cp`/CSDB routing gap noted in round 3/round 5 (¬ß notes above) ‚Äî anything that needs its own handler to see the raw prefix must bypass `wd_chat`.

**Fix:** `wedge_dispatch_body`'s `wd_first` now checks for a leading `:` right alongside the existing `#` check, before the keyword table scan even starts ‚Äî `cmp #$3a` / `bne wd_scan` / `jmp send_cli` (`rr38p-tmp12reu.bank05.asm`, `wd_check_pysic`). `send_cli` forwards the raw shadow line with no `"I:"` wrapper (same mechanism already used for `#c`/`#n`), so `PythonEvalHandler.can_handle()` (now `text.strip().startswith(":")`) sees its own prefix directly and finally gets a chance to claim the line ahead of `ChatHandler`.

**Also updated for the rename:** the exclusion lists in `chat_handler.py`, `csdb_handler.py`, `netdrive_handler.py` (`"?"` ‚Üí `":"`, so those handlers still yield to PySIC while their own module is active); `help_handler.py`'s static help text and topic body; `docs/user_manual/pysic.md` (+ its `cloud/static/docs/` mirror) and `cloud/static/docs/README.md`; `cloud/prompt.md`'s routing table; and `cloud/test_handlers.py`'s PySIC/cross-handler tests. No wedge keyword collision: `:` was not previously meaningful as a *leading* character anywhere in the wedge (confirmed ‚Äî only `#` was reserved there; `:` appears elsewhere only as the second character of two-letter prefixes like `I:`/`c:`/`m:`, which this check doesn't touch since it only fires on the first character of the line).
